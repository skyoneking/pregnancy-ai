## Context

当前项目是基于 Next.js App Router + LangChain/LangGraph 构建的天气助手。已有完整的流式对话架构、工具调用机制（含审批流）、GLM 模型接入和 MemorySaver 会话记忆。转型目标是将其改造为孕期助手 MVP，聚焦孕中阶段，服务夫妻双方。

**当前架构关键路径：**
```
前端 (page.tsx) → useAgentChat hook → POST /api/chat → agent.ts (LangChain) → 工具 → 流式响应
```

转型需要在此路径的每一层注入孕期上下文（用户身份 + 孕周），并替换业务工具集。

## Goals / Non-Goals

**Goals:**
- 新增 Onboarding 页面，收集用户档案（身份 + 预产期）并写入 localStorage
- 将孕期上下文（role + dueDate）从前端贯通至 LangChain agent context
- 实现五个孕期专用工具：孕周计算、每周胎儿发育、食物安全查询、产检时间表、四级症状评估
- 主聊天界面全中文化，显示孕周标签，移除天气相关 UI
- 为"传统应用 + AI 融合"的未来扩展保留结构化数据入口

**Non-Goals:**
- 服务端用户账户系统（MVP 仅用 localStorage）
- 孕前/孕后阶段功能
- 推送提醒、日历集成、数据可视化
- 真实医疗数据库接入（工具使用静态/模拟数据）
- 工具审批流（移除，孕期场景下无需审批延迟）

## Decisions

### 决策 1：用户档案存储在 localStorage（而非服务端）

**选择：** localStorage，key `pregnancy_profile`

**理由：**
- MVP 阶段无需用户注册/登录，降低开发成本
- 数据为个人隐私，存在本地反而更合适
- 结构简单：`{ role, dueDate, createdAt }`
- 未来扩展时，可将 localStorage 作为"客户端缓存层"，在同步服务端账户时复用

**未考虑的替代方案：** cookie（传输开销）、IndexedDB（对这种简单结构过度设计）

---

### 决策 2：孕期上下文通过 API 请求体传递（而非服务端 session）

**选择：** 每次请求携带 `{ messages, profile: { role, dueDate } }`

**理由：**
- 与当前无状态 API 设计一致（LangChain 用 thread_id 管理会话状态）
- 前端已有 userProfile，直接传入比服务端查询更简单
- 无需引入新的服务端状态管理

**风险：** 用户可篡改请求中的 profile，但 MVP 阶段可接受（无真实医疗决策依赖此数据）

---

### 决策 3：工具数据使用静态/规则数据（不接入真实医疗 API）

**选择：** 工具内置静态数据集（孕周发育、食物安全列表、产检时间表）

**理由：**
- 避免外部 API 依赖和费用
- 数据相对稳定，不需要实时更新
- 降低 MVP 复杂度，专注验证 AI 交互体验
- 未来可替换为真实 API 而不改变接口

**食物安全数据示例：**
```typescript
const FOOD_SAFETY: Record<string, { level: '安全' | '适量' | '避免' | '禁止', reason: string }> = {
  '螃蟹': { level: '避免', reason: '寒凉食物，可能引起子宫收缩' },
  '咖啡': { level: '适量', reason: '每日不超过200mg咖啡因' },
  '叶酸': { level: '安全', reason: '孕期必需营养素' },
  // ...
}
```

---

### 决策 4：症状评估采用四级分级，AI 自主判断

**选择：** 工具接收症状描述，由工具函数（结合规则）返回等级，AI 再补充解释

**四级定义：**
- 🟢 **正常**：孕期常见，无需担心
- 🟡 **观察**：值得留意，下次产检时告知医生
- 🟠 **就诊**：应在 24-48 小时内就医
- 🔴 **紧急**：立即前往急诊

**免责声明机制：** 工具输出中强制包含免责声明字段，系统提示要求 AI 必须展示

---

### 决策 5：移除工具审批流

**选择：** 孕期工具调用直接执行，无需用户审批

**理由：**
- 孕期场景中，延迟反馈（尤其是症状评估）会增加用户焦虑
- 当前审批流是为"高风险操作"（天气查询实际无风险，纯演示）设计
- 孕期工具为查询类，无副作用，不需要审批

---

### 决策 6：Onboarding 路由为独立页面 `/onboarding`

**选择：** `app/onboarding/page.tsx`，主页 `useEffect` 检测 localStorage 后跳转

**理由：**
- 与现有 App Router 结构一致
- 逻辑清晰：onboarding 是独立流程，不应嵌入主聊天页
- 未来可扩展为多步骤 onboarding（孕前档案、孕中档案等）

## Risks / Trade-offs

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| 静态医疗数据准确性 | 食物安全、产检时间表可能不适用于所有用户情况 | 每个工具输出附带"请以医生建议为准"的免责声明 |
| 症状评估误导 | AI 判断🟢正常，但实际需要就医 | 系统提示明确要求"宁严勿松"，不确定时提升等级；固定免责声明 |
| localStorage 数据丢失 | 用户清除浏览器数据后需重新 Onboarding | MVP 可接受；未来可提供数据导入/恢复功能 |
| thread_id 硬编码为 "1" | 多用户/多设备无法隔离会话历史 | MVP 阶段仅支持单设备使用，后续可改为基于 profile 生成唯一 thread_id |
| 孕周计算精度 | 仅凭预产期计算孕周，未考虑早产/个体差异 | 工具输出注明为"估算孕周" |
